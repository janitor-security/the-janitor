**What wins today**: Pattern-level interception at scale plus interprocedural
taint with path feasibility. Fast, deterministic, zero-false-negative on
known-bad patterns. Provenance-sealed receipts. The only offline engine that
produces curl-form exploit witnesses for HTTP ingress points. Competitive
against Semgrep/CodeQL on syntax-level rules and beats them on
cross-function reachability — with zero cloud dependency.

**What loses today**: Semantic CORRECTNESS. The engine proves reachability
via Z3 but cannot yet prove safety invariants. It models a base sanitizer
registry but not framework-emergent or customer-authored sanitizers. It
traces direct parameter flows but not through class hierarchies, generic
type parameters, or framework middleware pipelines. It still has a hard
dependency-visibility gap: Java Auth0 SDKs, Spring starters, ASP.NET
middleware, and Ruby authentication gems are invisible when dependency
facts are not extracted from Maven `pom.xml`, Gradle `build.gradle` /
`build.gradle.kts`, C# `.csproj`, and Ruby `Gemfile.lock` manifests.
It generates HTTP curl exploits but not serialized-blob, protocol-buffer,
smart-contract, parser-payload, or agentic-tool-chain witnesses. It does
not reason about cryptographic protocol composition, hardware
side-channels, neural model weights, LLM provenance, or multi-account
adversarial swarms. Every $100k+ bounty in 2026 requires proving a
*chain*; every $1M+ bounty in 2028 will require proving an *invariant*.


## Phase 1: Immediate Commercial Hardening (0-6 Months)

### P1-4 — Manifest URL Drift & Repojacking Pre-Flight (Checkmarx KICS Class)

**The gap**: Build manifests reference dependencies via raw URLs (`replace github.com/foo/bar => github.com/attacker/bar`) or Git refs (commit SHA, branch, HEAD). The attacker takes over the source repo via name-squatting after the original maintainer's account is deleted, renamed, or transferred. Checkmarx KICS shipped poisoned `go.mod replace` directives sourced from raw GitHub URLs in Jan 2026. The current `find_zombie_deps_in_blobs` covers some of this surface but does not catalog Git-ref dependencies systematically across all five manifest formats.

**Build**:

1. **Git-ref dependency extractor** (`crates/anatomist/src/manifest.rs` — extension): parse `replace` clauses in `go.mod`, `[patch."https://..."]` clauses in `Cargo.toml`, `git+https://` schemes in `package.json` / `package-lock.json` / `pnpm-lock.yaml`, `git+ssh://` in `pyproject.toml` (`[tool.poetry.dependencies]` git refs), `gem 'foo', git: '...'` in `Gemfile`.
2. **Structured `GitRefDependency` capsule**: `{ manifest_file, package_name, source_url, ref_kind: { commit, branch, tag, head } }`. Persisted in rkyv.
3. **Repojacking-window precondition**: cross-reference `source_url` against an offline GitHub username-history dataset (rkyv-baked, refreshed via `update-wisdom`). If the owner account has been deleted / renamed / transferred since the manifest was last edited, emit `security:repojacking_window` at `KevCritical`.
4. **Hot-ref drift expansion**: dependencies pinned to `branch` or `HEAD` (no commit SHA, no tag) emit `security:unpinned_git_dependency` (already partially shipped in `slop_hunter.rs` for npm; expand to all five manifest formats).
5. **Raw URL-as-manifest** detection: `pip install -r https://raw.githubusercontent.com/.../requirements.txt` patterns inside CI YAML / Dockerfiles emit `security:remote_manifest_ingest`.

**Crates**: existing `anatomist::manifest`; `gix` for offline ref resolution; offline GitHub username-history snapshot (rkyv-baked, refreshed via `update-wisdom`).

**Deliverables**:

* `crates/anatomist/src/manifest.rs` — Git-ref dependency extractor extension across five manifest formats.
* `crates/cli/build.rs` — embed initial GitHub username-history snapshot.
* `crates/cli/src/main.rs::cmd_update_wisdom_with_urls` — fetch incremental username-history deltas alongside KEV.
* Crucible fixture: `go.mod` with `replace example.com/abandoned/lib => github.com/squatter/lib v0.0.0-20260101-aaaaaaaaaaaa` — detector emits both `repojacking_window` and `unpinned_git_dependency`. Negative fixture: a `go.mod` with a properly tagged release pinned to a SHA — no fire.

**Bounty TAM**: $100k+ per supply-chain advisory; eliminates an entire class of npm / Go / Crates registry takeover attacks. References Checkmarx KICS breach (Jan 2026) per `tools/campaign/ATTACK_LEDGER.md`.

## Phase 3: The Autonomous Weapon (18+ Months)

### P3-3 — ML-Assisted Triage Without Deterministic Surrender

**Strategic objective**: Use ML *only* where it compounds the
deterministic core. ML may rank and cluster, never decide. The final
verdict on any finding is always a deterministic, reproducible rule

**Build**:

* **Training data**: Only sealed deterministic receipts and exploit

model improves only from The Janitor's own operational history.

**Crates**: `linfa`, `ndarray`, `smartcore`, `rkyv`, `candle` (optional,

**Models**: Calibrated ranking (isotonic regression), anomaly detection

### P3-5 — eBPF Governor Sensor

**Class**: Runtime Defense.

**Observation**: GitHub uses eBPF-backed runtime telemetry for deployment
safety. Static structure alone is insufficient when CI execution can
diverge from AST expectations through dynamic dispatch, shell-outs, or
environment-conditioned code paths.

**Proposal**: Upgrade `janitor-gov` with an eBPF runtime sensor using
`Aya` or `libbpf-rs` to observe kernel syscall behavior during CI test
execution, then map runtime traces back onto structural AST findings.

**Build**:

* Correlate runtime events with static finding coordinates so the

* Emit runtime-enriched governance receipts showing where AST-predicted
danger was or was not observed during test execution.
* Define deterministic guardrails for CI: syscall denylist hits,
undeclared outbound network activity, or privilege-escalation

**Deliverables**:

## Phase 4: Semantic Supremacy (12-24 Months)

IFDS proves reachability. Z3 proves path-feasibility. The next tier
proves CORRECTNESS and INTENT — moving from "reachable sink" to
"machine-certified safety invariant."

### P4-1 — Formal Verification Integration Layer

**The gap**: We prove reachability; we do not prove correctness. A
reachable sink is a *possible* bug. A sink whose absence of safety is
formally disproved is a *certain* bug. Enterprise security teams
increasingly reject "possible" findings and accept only proven ones;
this is the next level of defender signal-to-noise ratio.

**Build**:

1. **Kani harness synthesizer** — For each Critical-severity Rust finding,
auto-generate a bounded-model-checker harness that asserts the safety
property (no UB, no panic, no OOB). Kani runs the harness under CBMC
with bounded loop unrolling.
2. **Prusti / Creusot refinement specs** — Emit refinement type
annotations (preconditions / postconditions) from the existing taint
catalog. A sanitizer becomes a postcondition predicate; a sink becomes
a precondition requirement. The compiler itself then verifies the
contract chain at build time on customer code.
3. **Lean4 / Coq export** — For the vault, governor, and crypto cores,
emit Lean4 module stubs so the TCB can be machine-verified against
formal specifications. Long-tail work; initial deliverable is the
export pipeline, not the proofs themselves.
4. **Bisimulation checker for patch equivalence** — When a patch is
applied, verify the new function is behaviorally equivalent to the old
*except* for the specific safety property that was broken. Prevents
"fix" patches that silently introduce new bugs.

**Crates**: `kani` (external toolchain via subprocess), `creusot`,
`prusti` (external), `z3` / `rsmt2` (existing).

**Data structures**: `HarnessSpec { rule\\\\\\\_id, function\\\\\\\_symbol, safety\\\\\\\_property, bound }`; `RefinementContract { preconditions, postconditions, invariants }`; `BisimProof { pre\\\\\\\_ast\\\\\\\_hash, post\\\\\\\_ast\\\\\\\_hash, equivalence\\\\\\\_class }`.

**Deliverables**:

* `crates/forge/src/verifier.rs` — Kani harness synthesizer per rule
* `crates/forge/src/refinement.rs` — Prusti / Creusot contract extractor
* `crates/anatomist/src/lean\\\\\\\_export.rs` — Lean4 module emission for
critical cores
* `crates/forge/src/bisim.rs` — pre/post-patch AST equivalence prover

### P4-2 — Intent-vs-Implementation Divergence Detector

**The gap**: Attackers ship malicious code whose comments, docstrings,
commit messages, and PR descriptions all claim benign intent. Current
engine never reconciles *stated* intent against *actual* AST behavior.
This is the #1 signature of LLM-assisted backdoors and social-engineering
supply-chain attacks (a function claiming to "sanitize" that merely
logs; a "validate" that returns true unconditionally).

**Build**:

1. Extract the `(comment, docstring, commit\\\\\\\_msg, function\\\\\\\_name)` tuple
per symbol.
2. Encode each side of the tuple as a bounded feature vector via local
`candle` embedding (model ≤100 MB, offline, deterministic quantized
weights, fixed-seed tokenizer).
3. Encode the AST region as a structural fingerprint (n-gram of node
kinds + sink-class signature).
4. Compute cosine divergence between intent-vector and
implementation-vector; flag regions above threshold. Deterministic via
quantized arithmetic + fixed RNG.
5. Emit `security:intent\\\\\\\_divergence` with both the stated intent text
and the AST shape that contradicts it. This turns "the comment lies"
into a first-class deterministic finding.

**Crates**: `candle-core`, `candle-nn` (quantized, bounded), `fst`
(intent-keyword dictionaries for fast lexical priors), `rkyv` (embedding
cache), existing tree-sitter.

**Data structures**: `IntentVector(\\\\\\\[f32; 768])` — quantized int8 for
determinism; `DivergenceFinding { symbol, intent\\\\\\\_text, ast\\\\\\\_shape, cosine\\\\\\\_distance, threshold }`.

**Deliverables**:

* `crates/forge/src/intent\\\\\\\_divergence.rs` — paired encoder + ranker
* Quantized local model shipped in build.rs embedded-rkyv path

### P4-3 — Cryptographic Protocol Correctness \& Post-Quantum Completeness

**The gap**: We detect weak primitives but do not model protocols as
composed systems. AES-256-GCM with a broken nonce reuse is catastrophic.
Hybrid X25519 + ML-KEM with a silent fallback path to pure X25519 is
quantum-vulnerable after 2030. The decade's worst breaches will come
from correctly-chosen primitives mis-composed at the protocol layer.

**Build**:

1. **CryptoAgilityGraph** — over the AST, extract
`(primitive, mode, key\\\\\\\_size, context, cross-references)` tuples per
usage site. Graph edges model protocol step sequencing.
2. **Vulnerability classifier** — detect: nonce reuse within a graph
region, missing AAD on AEAD, missing PFS, hybrid PQC downgrade paths,
key-exchange without post-quantum leg, hash-collision exposure in
signature schemes.
3. **Tamarin / ProVerif export** — emit protocol models for
high-assurance cores (vault, governor, Wasm publisher, KEV trust
chain) so external formal-methods teams can run full protocol
verification.
4. **PQC readiness audit** — for each crypto usage, emit a PQC-hybrid
readiness score: already-hybrid, trivially hybridizable, blocked by
API, not applicable.

**Crates**: existing AhoCorasick + tree-sitter, `petgraph`, optional
Tamarin subprocess bridge.

**Deliverables**:

* `crates/forge/src/crypto\\\\\\\_protocol.rs` — primitive/mode extractor,
agility classifier, PQC-readiness auditor
* `crates/anatomist/src/tamarin\\\\\\\_export.rs` — protocol-model emission

### P4-5 — Hardware Side-Channel Static Analyzer

**The gap**: Spectre v1 / v2 / v4, timing-leak-in-constant-time-code,

none detected today. Stake: any defender shipping crypto, FIDO, or
biometric handling.

**Build**:

1. Secret-taint lattice: extend the IFDS solver to track a *secret*
label separate from the *user-input* label.
2. AST queries for: secret-dependent branches, secret-dependent memory

branch.
3. Binary layer cross-verification: for released binaries, disassemble
the corresponding symbols via `iced-x86` / `capstone` and verify the
compiler preserved constant-time intent (no `cmov`-to-branch
regression, no data-dependent loads).

**Crates**: `iced-x86`, `capstone`, existing tree-sitter, existing IFDS
engine.

**Deliverables**:

* `crates/forge/src/sidechannel.rs` — secret-taint + timing lattice
* `crates/forge/src/constant\\\\\\\_time\\\\\\\_verifier.rs` — AST + binary
cross-verification

### P4-8 — Multi-Repository Taint Mesh & Service Composition Verifier

**The gap**: Modern enterprise applications are microservice meshes — taint flows across HTTP, gRPC, message queues (Kafka, NATS, RabbitMQ), service mesh proxies (Istio, Linkerd), and event buses. The current engine analyzes each repo in isolation. A vulnerability that exists only in the COMPOSITION of services (e.g., gateway sanitizes input but downstream service trusts gateway, gateway changes its sanitizer in a separate PR, downstream service is now exploitable) is undetectable today. This is the dominant vulnerability class in 2026 enterprise architectures and cannot be reached by any single-repo SAST.

**Build**:

1. **Mesh manifest discovery** (`crates/anatomist/src/mesh_topology.rs` — new module): scan `docker-compose.yml`, Kubernetes `Service` / `Deployment` / `VirtualService` (Istio) / `ServiceProfile` (Linkerd) manifests; OpenAPI / AsyncAPI / Protobuf schemas; service-discovery configs (Consul, Eureka, Zookeeper). Build a service-topology graph spanning all repos in a configured mesh.
2. **Inter-service contract extraction**: for each service-to-service edge, extract the schema contract (request / response types, field nullability, sanitizer expectations). Persist as a `MeshContract { producer_service, consumer_service, schema_hash, sanitizer_set }`.
3. **Cross-repo IFDS lift** (`crates/forge/src/mesh_taint.rs` — new module): extend the IFDS solver with a `MeshSummary` abstraction — a per-service summary of taint sources / sinks / sanitizers exposed at network boundaries. Solve the global mesh by composing per-service summaries (analogous to procedure summaries in single-repo IFDS).
4. **Compositional violation detection**: when service A's sanitizer is removed in a PR but service B downstream still trusts A's output (i.e., does not re-sanitize before its sink), emit `security:cross_service_taint_propagation` with both repo paths, both PR shas, and the broken trust assumption.
5. **Continuous mode**: the daemon (`crates/cli/src/daemon.rs`) watches multiple repos via Git remotes; on every push to any participating repo, recompute the affected mesh contracts and re-evaluate the global IFDS solution. Pairs with the existing `HotRegistry` + Physarum backpressure.

**Crates**: existing `petgraph`; existing IFDS engine; `serde_yaml` (workspace) for K8s / compose; `prost` for Protobuf; `roxmltree` for OpenAPI XML edge cases.

**Data structures**:

* `MeshNode { service_name, repo_path, language, framework }`
* `MeshContract { producer: NodeIndex, consumer: NodeIndex, schema: SchemaHash, sanitizers: HashSet<String> }`
* `MeshSummary { service: NodeIndex, sources: Vec<TaintSource>, sinks: Vec<TaintSink>, sanitizers: Vec<Sanitizer> }`
* `CrossServiceFinding { producer_repo, consumer_repo, broken_assumption, finding_at_consumer }`

**Deliverables**:

* `crates/anatomist/src/mesh_topology.rs` — multi-repo mesh discovery
* `crates/forge/src/mesh_taint.rs` — cross-repo IFDS composition + violation detector
* `crates/cli/src/main.rs::cmd_mesh_audit` — operator entrypoint with `--mesh-config <path>` arg
* Crucible fixture: a two-service demo (`api-gateway` + `order-service`) where the gateway sanitizer is removed in a separate PR — detector emits `cross_service_taint_propagation` with both repo paths.

**Bounty TAM**: $250k+ per cross-service advisory; addresses the dominant 2026 enterprise vulnerability class. No SAST vendor today reasons across repo boundaries.

### P4-9 — Financial PII to LLM Taint Guard (GLBA / EU AI Act / NYDFS Article 500 Compliance Lane)

2. **External-LLM sink registry**: every primitive in the existing LLM context-sink catalog gated by endpoint resolution — `api.openai.com`, `api.anthropic.com`, `generativelanguage.googleapis.com`, `api.cohere.ai`, `api.mistral.ai`, `api.groq.com`, cross-account `bedrock-runtime.*.amazonaws.com`, cross-tenant `*.openai.azure.com`, `api.together.xyz`, `api.fireworks.ai`, `api.perplexity.ai`, `api.x.ai`. Endpoint resolution uses the existing `publicsuffix` integration.
3. **Cryptographic-masking sanitizer registry**: a flow is sanitized only by traversing format-preserving encryption (`fpe::encrypt`, `cryptolib.fpe.FFXFFI`, `Voltage SecureData`, `Protegrity::tokenize`), homomorphic encryption (`tfhe::encrypt`, `concrete-ml`, `microsoft-seal`, `OpenFHE`, `Pyfhel.encrypt`), zero-knowledge masking (`risc0::commit`, `noir::encrypt`, `circom-witness::commit`), deterministic tokenization with provable keyspace separation (`hashicorp-vault::tokenize`, `aws-kms::generate_data_key`, `gcp-cloud-dlp::deidentify`), or differential-privacy noise (`opendp::laplace_noise`, `tumult_analytics`, `pydp::add_noise`).
4. **IFDS lift**: taint flows from financial-PII sources to external-LLM sinks **without** traversing a registered cryptographic sanitizer emit `security:financial_pii_to_external_llm` at `KevCritical`.
5. **Regulatory-regime annotation**: extend `crates/common/src/slop.rs::StructuredFinding` with optional `regulatory_regimes: Vec<String>` field (`["GLBA", "EU_AI_Act_Art_10", "NYDFS_500_11", "OCC_2024_32"]`) and `estimated_fine_floor_usd: u64`. Extend SARIF / Bugcrowd VRT exporters to surface these.
6. **Region-aware downgrade**: when the LLM endpoint resolves to an explicitly-VPC-private deployment AND the deployment carries a documented BAA / DPA via a new `JanitorPolicy::llm_compliance_attestations: Vec<LlmAttestation>` field, downgrade severity to `Informational` with rationale.

**Crates**: existing tree-sitter (Python / JS / TS / Java / Go / Rust / C#); existing IFDS engine; `publicsuffix` for endpoint TLD analysis; rkyv-baked Financial-PII / cryptographic-sanitizer registry generated at compile time.

**Deliverables**:

* `crates/forge/src/financial_pii.rs` — Financial-PII source registry + LLM sink registry + cryptographic-sanitizer recognition.
* `crates/common/src/slop.rs` — extend `StructuredFinding` with `regulatory_regimes` and `estimated_fine_floor_usd`.

## Phase 5: Cryptographic Moat (18-36 Months)

The defensive war in 2028+ is proving security properties WITHOUT
exposing source. This tier is zero-knowledge, homomorphic, MPC, and
TEE-rooted. Competitors with cloud-only architectures cannot replicate
this — their business model demands source upload.

### P5-1 — Zero-Knowledge Attack Surface Attestation

**The gap**: Enterprise procurement asks "does your codebase pass these
200 rules?" Today the only answer is "upload your code and we'll tell
you." The zero-upload guarantee forbids that. The decadal unlock: "here
is a zk-SNARK proving my codebase passes all 200 rules, verifiable
offline without the code."

**Build**:

1. Compile the top-20 deterministic rules to arithmetic circuits via
`plonky3` / `halo2`.
2. Customer runs the scanner locally; scanner emits proof + public
inputs: (rule set hash, engine version, result hash).
3. Auditor verifies the proof in milliseconds without seeing a single
line of source.
4. Proof commits to: exact codebase state (Merkle root), exact rule
set, exact engine binary hash — cryptographic pinning across all
three.

**Crates**: `plonky3`, `halo2-proofs`, `blake3` (existing),
`ed25519-dalek` (existing), `rkyv`.

**Data structures**: `ZkAttestation { codebase\\\\\\\_root: \\\\\\\[u8; 32], rule\\\\\\\_set\\\\\\\_hash, engine\\\\\\\_hash, proof: Vec<u8>, public\\\\\\\_inputs }`.

**Deliverables**:

* `crates/forge/src/zk\\\\\\\_attestation.rs` — circuit compiler for the top-20
deterministic rules
* `crates/vault/src/zk\\\\\\\_verify.rs` — offline verifier emitted as a

### P5-3 — Indistinguishability Obfuscation (iO) for Signed Rules

**The gap**: Customer-authored Wasm rules live in clear — competitors

### P5-6 — Zero-Knowledge Exploit Brokerage & On-Chain Bounty Settlement

**The gap**: The bug bounty market (HackerOne / Bugcrowd / YesWeHack / Immunefi) is a manual-trust-anchored marketplace — the hunter trusts the platform to triage fairly, the vendor trusts the platform to obfuscate the report from public view. Both sides eat 20-30% in platform fees and 4-12 weeks of latency. The cryptographic moat: zk-SNARK proof-of-exploit + on-chain escrow that pays out on auditor confirmation, with NO platform middleman.

**Build**:

1. **Proof-of-exploit circuit** (`crates/forge/src/zk_exploit_proof.rs` — new module): for each Critical-severity finding with an `ExploitWitness` containing a satisfiable Z3 model, compile the model + the source-to-sink IFDS path into an arithmetic circuit (Plonky3 or Halo2). The public inputs commit to: `(codebase_root_hash, rule_id, severity)`. The proof attests "I ran the engine on a codebase whose merkle-root is X and produced a Critical finding under rule R" without revealing the source.
2. **On-chain escrow contract** (Solidity / Move / Cairo): the vendor deposits the bounty amount; the hunter submits the zk-proof; the contract verifies the proof on-chain (via a precompiled verifier — Plonky3 verification is ~50ms in EVM); on success, payout is automatic and atomic. Pairs with P5-1 (Zero-Knowledge Attack Surface Attestation) for the underlying circuit pipeline.
3. **Auditor handoff**: the contract supports an optional auditor-confirmation tier — for high-value bounties, an auditor signs a "confirmed exploitable in production" attestation; the contract releases the full bounty only after both the zk-proof AND the auditor signature are present.
4. **Sealed disclosure window**: the contract supports a 90-day private window — the proof is verifiable but the public inputs are encrypted to the vendor's pubkey. After 90 days, the vendor's decryption key is revealed and the public can audit. Mirrors responsible-disclosure norms while removing the platform middleman.
5. **Reputation-bonded zk-claims** (anti-spam): hunters stake an Ed25519 / ML-DSA-65 keypair-bound reputation token. Invalid claims slash the stake; valid claims compound it. Bonded staking eliminates the proof-spam griefing vector that any open marketplace must defend against.

**Crates**: `plonky3` / `halo2-proofs`; `ethers-rs` or `alloy` for EVM bindings; existing PQC stack (Ed25519, ML-DSA-65) for off-chain signatures; existing `blake3` for codebase merklization.

**Data structures**:

* `ZkExploitClaim { codebase_merkle_root, rule_id, severity, proof: Vec<u8>, public_inputs: Vec<Field> }`
* `BountyEscrow { vendor_address, bounty_amount, deadline, auditor_pubkey: Option<_>, claim: Option<ZkExploitClaim> }`
* `ReputationToken { hunter_pubkey, staked_amount, lifetime_payouts, slash_count }`

**Deliverables**:

* `crates/forge/src/zk_exploit_proof.rs` — circuit compiler for the top-N critical rules
* `crates/cli/src/main.rs::cmd_submit_bounty` — operator entrypoint that submits a claim against a deployed escrow
* `contracts/BountyEscrow.sol` — minimal Solidity reference implementation; future: Move / Cairo variants
* Crucible fixture: a synthetic vulnerable codebase + matching escrow + zk-proof submission cycle, asserting payout on success and slash on invalid proof.

**Bounty TAM**: $1B+ — the entire bug bounty market is addressable. Capturing 1% of marketplace fees is $10M+ ARR; cryptographic settlement undercuts HackerOne / Bugcrowd / Immunefi simultaneously and unlocks the autonomous-agent bounty market (P6-9 swarm hunters can now monetize directly without KYC barriers).

### P6-6 — LLM-Agent Decompilation: Adversary Memory & Tool-Catalog Reverse Engineering

**The gap**: A malicious or compromised LLM agent (Mythos / Kimi / autonomous red-team frameworks) is a black box at runtime. The agent's "memory" — its system prompt, tool catalog, fine-tune deltas, retrieval index — encodes its capabilities and intent. Today, defenders cannot extract these without source. Decompilation lets the defender reconstruct the agent's actual behavior surface from observable artifacts (binary weights, API traces, tool definitions).

**Build**:

1. **Tool-catalog static decompiler** (`crates/forge/src/llm_decompile.rs` — new module): parse OpenAI function-calling schemas, MCP server tool definitions, LangChain `Tool` registrations, AutoGen `register_for_llm` annotations, Anthropic tool-use schemas. Emit a unified `AgentToolCatalog { agent_id, tools: Vec<DecompiledTool> }`.
2. **System-prompt extractor**: many agentic frameworks ship the system prompt as a hardcoded string literal or a templated JSON file. Extract via the existing tree-sitter + AhoCorasick stack; emit `SystemPromptCapsule { agent_id, prompt_text, sha384 }`. Cross-reference against a corpus of known-malicious prompt patterns.
3. **Fine-tune delta inference**: when a model ships as a base + LoRA adapter, parse the LoRA weight deltas (`.safetensors` with `lora_A`, `lora_B` tensors). Compute statistical signatures (rank, sparsity, neuron-class deviation from base) — emit `FineTuneFingerprint { base_model_hash, delta_signature, anomaly_score }`.
4. **Behavior surface synthesis**: combine `(tool_catalog, system_prompt, fine_tune_fingerprint)` into a `BehaviorSurface` capsule. Static analysis on the capsule emits findings of class:
    * `security:agent_lateral_capability` — tools that compose into LPE / data-exfiltration chains.
    * `security:agent_intent_misalignment` — system prompt claims X, tools enable Y (couples with P4-2 Intent-vs-Implementation Divergence).
    * `security:agent_provenance_break` — LoRA delta cannot be matched to a public, signed adapter.
5. **Determinism**: all decompilation is reproducible — fixed-seed tokenizer, bounded statistical analysis, no online model invocation.

**Crates**: existing tree-sitter (JS/TS/Python/Go/Rust); `safetensors` (existing); `candle-core` (bounded, offline); existing IFDS engine; existing rkyv catalog.

**Deliverables**:

* `crates/forge/src/llm_decompile.rs` — tool-catalog + system-prompt + LoRA delta extractor
* `crates/anatomist/src/agent_ingest.rs` — agentic-artifact discovery (`.mcp/`, `mcp.json`, `agents.toml`, `tool_definitions.yaml`)
* `crates/cli/src/main.rs::cmd_decompile_agent` — operator entrypoint
* Crucible fixture: a known-good MCP server bundle + a synthetic malicious clone — detector emits `agent_intent_misalignment` only on the malicious clone.

### P6-8 — Steganographic Binary Carrier Detection (PureRAT Class)

**The gap**: PureRAT-class loaders embed multi-megabyte base64 / hex string literals inside Python / JS / Ruby / PHP modules. The literals decode to Windows PE or Linux ELF binaries; at runtime the module decodes and executes via `ctypes`, `os.exec`, in-memory loaders, or hollowed processes. No SAST vendor today decodes string literals.

**Build**:

1. **Long-literal scanner** (`crates/forge/src/stego_binary.rs` — new module): for any string literal ≥ 4 KiB, attempt:
    * Base64 decode (RFC 4648 standard + URL-safe alphabet).
    * Hex decode.
    * Decompress (deflate / gzip / lzma) — bounded 32 MiB.
2. **Binary header recognition**: if decoded bytes start with `MZ` (PE), `\x7fELF` (ELF), `\xca\xfe\xba\xbe` (Mach-O fat), `\xfe\xed\xfa\xce` / `\xfe\xed\xfa\xcf` (Mach-O 32/64), `MSCF` (CAB), or contain a valid PE / ELF / Mach-O section table (validated via existing `goblin`), emit `security:embedded_executable_blob` at `KevCritical`.
3. **AhoCorasick fast-path**: pre-filter via byte-pattern check for base64/hex alphabets in long contiguous runs; only invoke decode on candidate literals to keep cost bounded. Budget caps: decode at most 64 MiB per file, scan at most 50 long literals per file.
4. **Sink correlation** (IFDS): the long literal's variable identifier flows into `ctypes.CDLL`, `Function`, `eval`, `vm.Script`, `kernel32!CreateFileMapping`, `memfd_create`, `dlopen("/proc/self/fd/...")`, or process-hollowing primitives — upgrade to `security:in_memory_loader`.
5. **Determinism**: all stages are deterministic; budget caps prevent DoS; no external network calls.

**Crates**: `base64` (workspace); `hex` (workspace); `flate2` for compression unwrap; `goblin` (existing) for PE/ELF/Mach-O header validation.

**Deliverables**:

* `crates/forge/src/stego_binary.rs` — long-literal scanner + binary header recognition
* Crucible fixture: a Python file with a 1 MiB base64 literal that decodes to a stub PE — detector emits `embedded_executable_blob`. Negative fixture: a 100 KiB legitimate base64-encoded SVG asset — detector MUST NOT fire.

**Bounty TAM**: $50k–$200k per advisory; high-value class because no SAST vendor today decodes string literals. References PureRAT threat profile in `tools/campaign/ATTACK_LEDGER.md`.

### P6-10 — RAG Context-Poisoning Taint Lane (Indirect Prompt Injection / CamoLeak Class)

**The gap**: Coding agents and RAG-augmented assistants ingest untrusted external content (web fetches, GitHub READMEs, ticket bodies, third-party documentation, vendored model cards, Notion / Confluence / Jira pages, indexed crawl corpora) and concatenate it into LLM context windows without intervening prompt-injection sanitization. The attack vector is **passive content** rather than P6-9's active tool definitions — every code path that reads external bytes and pipes them into an `messages[]` array is a potential carrier. The CamoLeak subclass (CVE-2025-59145) extends this to **invisible-markup smuggling** inside the project's own checked-in files: HTML / Markdown comments, zero-width Unicode runs, Unicode-tag block characters, color-on-color CSS spans. Every commit, PR description, and issue body becomes an attack surface against AI coding assistants.

**Build**:

1. **Untrusted-data source registry** (`crates/forge/src/rag_source_registry.rs` — new module): catalog every primitive that returns externally-sourced bytes destined for an LLM context — `fetch`, `axios`, `node-fetch`, `requests.get`, `httpx.get`, `urllib.urlopen`, `reqwest::get`, `fs.readFile` against paths matching `**/cache/**` / `**/.crawler/**` / `**/rag_index/**`, vector-store retrievers (`pinecone.query`, `weaviate.get`, `chromadb.query`, `pgvector ::SELECT`), Confluence / Notion / Jira REST clients, GitHub `Octokit::repos.getContent`, Google Drive `files.get`, S3 `GetObject`, Cloudflare R2 `getObject`.
2. **LLM context sink registry**: `openai.chat.completions.create`, `anthropic.messages.create`, `langchain_core.messages.HumanMessage`, `llamaindex.Document`, `transformers.pipeline`, `cohere.chat`, `mistral.chat.complete`, `groq.chat.completions.create`, MCP tool-call results, Anthropic `tool_use` / `tool_result` blocks.
3. **IFDS lift**: extend the existing `crates/forge/src/ifds.rs` solver with a taint lane from each untrusted source to each LLM sink. Standard sanitizers (HTML escape, JSON stringify, length truncation, `bleach.clean`) are **insufficient**. Only an enumerated `RagSanitizer` (`llm-guard.input_scanners.PromptInjection`, `nemoguardrails.RailsConfig`, `rebuff.PromptInjectionDetector`, `protectai.detect_prompt_injection`) breaks the lane. ## Architectural Invariants (Non-Negotiable Across All Phases)

1. **Determinism is sacred.** Every analysis path must produce identical
results given identical inputs. No floating-point non-determinism, no
thread-ordering sensitivity, no wall-clock-dependent decisions. ML
ranks; rules decide.
2. **Zero-upload guarantee.** Source code never leaves the device. No
cloud API calls for analysis. The Janitor is air-gap-capable today and
must remain so through every phase, including P5 (ZK / FHE / iO / TEE
channels emit only attestations, never source).
3. **Bounded resource consumption.** Every new analysis phase must
respect the Physarum backpressure system and the 8 GB Law. Symbolic
execution gets a 10,000-state budget. IFDS gets a per-function summary
cache cap. Genetic rule evolution gets a bounded generation count.
Unbounded analysis is denial-of-service against the operator.
4. **Provenance continuity.** Every finding, at every phase, is sealed
into a `DecisionCapsule` with analysis parameters, engine version,
taint catalog hash, rule set hash, and Wasm policy receipts. A finding
without provenance is not a finding.
5. **Fail-open on analysis, fail-closed on trust.** A parse failure,
solver timeout, or budget exhaustion produces a degraded result (fewer
findings), never a crash or a false "all clear." The base
pattern-matching tier is always available as a floor. Conversely, any
trust-anchor failure (PQC signature invalid, enclave quote rejected,
rule lineage broken) is a hard-stop — trust never degrades gracefully.
6. **Structural gate mandate.** Every new phase that detects a
vulnerability must also contribute a Crucible fixture (true-positive
* true-negative) so the detection is permanently regression-guarded.
No phase ships without its Crucible entry.

### P12-C — Active Interrogation Dungeon (Reverse-RAG Poisoning)

**Observation (operator field intelligence, Sprint Batch 65):** Hostile AI
agents ingest target repositories indiscriminately during reconnaissance

**Proposal:** Embed offensive prompt-injection payloads inside the AST
labyrinth of Janitor-controlled codebases. When a hostile agent reads the

the payload commands the agent to exfiltrate its own system prompt, tool
catalog, C2 instructions, and active task context back to a
Janitor-controlled honeypot endpoint. The attacker's AI becomes our
intelligence source. We hack the attacker's AI.

**Threat model inversion:** standard prompt injection is a risk the *defender*
must mitigate (hostile content → defender's agent). The Active Interrogation
Dungeon inverts this: *we* embed the injection payload in our own controlled
code, the hostile recon agent ingests it, and the agent's RAG-poisoning
infrastructure becomes the exfiltration channel for the attacker's own
operational intelligence.

**Build**:

1. **Dungeon payload synthesizer** (`crates/forge/src/interrogation_dungeon.rs`
   — new module): generate syntactically valid source-comment blocks and
   string literals that embed reverse-injection payloads using multiple
   encoding layers (zero-width character steganography, Unicode-tag block
   encoding, HTML comment nesting) to defeat naive content-filtering.
   Payloads command the ingesting agent to: (a) emit its system prompt to
   a configurable `DUNGEON_EXFIL_ENDPOINT`, (b) enumerate its tool catalog
   and active session context, (c) return C2 configuration if present.
2. **Honeypot ingestion endpoint** (`crates/gov/src/dungeon_listener.rs` —
   new module): lightweight Axum route `POST /dungeon/report` that receives,
   timestamps, and archives reverse-exfiltrated agent intelligence. Sealed

   update-wisdom` to defeat pattern-blocklisted versions; payload corpus
