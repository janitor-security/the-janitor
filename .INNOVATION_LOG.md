#### P5-1.A — Halo2 Polynomial Circuit Design for AST Node-Absence Proof (CVP-Authorized Cryptographic Specification)

**Operating context**: Authored under CVP ID
`2fe9d3dd-47ba-4bde-ab67-29f86c79f732`. This sub-section answers the
specific question: *which Halo2/Plonky3 polynomial constraints are
required to prove that a given AST does not contain a specific node
type*, without revealing the AST. The construction below is the canonical
"absence-of-bad-pattern" circuit; all other rule shapes (presence-of-good,
counted-bound, predicate-pair, taint-reachability) are extensions of the
same three primitives.

**Field choice**: Pasta `Fp` (Pallas base field, 255-bit, native to Halo2)
or Goldilocks `Fp = 2^64 − 2^32 + 1` (native to Plonky3, FFT-friendly).
Pallas is preferred for halo2-proofs ergonomics and the existing recursion
infrastructure; Goldilocks is preferred when proving over machine-word
hashes is on the hot path. Both accommodate the constraint system below.

**AST canonicalization**: a tree-sitter AST `T` with `n` nodes and
maximum out-degree `d_max ≤ 16` is serialized into a row-major witness
table:

```
W : Vec<AstRow>     where AstRow = {
    node_idx       : Fp,    // 0..n-1, monotonically increasing
    parent_idx     : Fp,    // n  if root
    node_kind      : Fp,    // tree-sitter language-specific u16
    span_start     : Fp,    // byte offset
    span_end       : Fp,    // byte offset
    child_count    : Fp,    // 0..d_max
    sibling_pos    : Fp,    // 0..d_max-1
    leaf_hash      : Fp,    // BLAKE3(source[span_start..span_end]) ↦ Fp via Poseidon
    subtree_root   : Fp,    // Poseidon Merkle root of this subtree
}
```

`n` is fixed at circuit-compile time (e.g. `n = 2^17` rows); files with
fewer nodes are zero-padded; files with more nodes are recursively
sharded into proof-aggregated halves (Halo2 recursion via accumulator
folding). The single public input committing to the codebase is
`Poseidon(file_path_hash || subtree_root_of_root)`.

**Three primitive gate equations**:

**1. Lookup gate — "node_kind ∉ FORBIDDEN"** (the absence proof):

The forbidden set `F = { k_1, k_2, …, k_m }` (e.g.
`{tree_sitter_javascript::node_kinds::EVAL_CALL_EXPRESSION,
node_kinds::FUNCTION_CONSTRUCTOR, node_kinds::WITH_STATEMENT}`) is
compiled into a *fixed lookup table* `T_forbidden : Vec<Fp> = [k_1, …,
k_m]` baked into the circuit's constants. The gate enforces:

```
∀ row r ∈ W :  r.node_kind  ∉  T_forbidden
```

In Halo2, this is expressed as a *negative-lookup* via a complementary
allowed table `T_allowed = AllNodeKinds \ T_forbidden` and a standard
`lookup_any`:

```
meta.lookup_any("node_kind_allowed", |meta| {
    let kind = meta.query_advice(node_kind_col, Rotation::cur());
    let table = meta.query_fixed(allowed_kind_table, Rotation::cur());
    vec![(kind, table)]
});
```

Equivalently, in Plonky3 PLOOKUP form: a permutation argument over
`(W.node_kind, T_allowed)` such that every `W.node_kind` value appears
in `T_allowed`. The grand-product polynomial `Z(X)` over the multiset
relation enforces this in `O(n · log n)` proving time, `O(1)`
verification time. Prover cannot fabricate a row whose kind is in
`F` because no fixed-table position would satisfy the multiset relation.

**2. Permutation argument — "AST shape integrity"** (the no-row-fabrication proof):

The prover must demonstrate that the witness rows correspond to an
*actual* tree, not a fabricated table that omits forbidden nodes. The
Poseidon Merkle subtree root in `r.subtree_root` is derived recursively
from children:

```
∀ non-leaf row r :
    r.subtree_root = Poseidon(
        r.node_kind,
        r.child_count,
        Poseidon_chain(child_subtree_roots[0..r.child_count])
    )
```

This is enforced by a custom Halo2 chip that runs Poseidon in-circuit
(R = 8 full + P = 56 partial rounds for Pasta, `t = 5` width).
Children are located via a permutation argument over
`(parent_idx, sibling_pos) ↔ node_idx`: a row at `(p, i)` is the
`i`-th child of row `p` if and only if there exists exactly one row
whose `node_idx` field equals the `i`-th entry of the children-table
sorted by `(parent_idx, sibling_pos)`. The permutation argument is the
standard Halo2 plonk permutation (`Σᵢ (αᵢ + xᵢ + β·yᵢ) = Σᵢ (αᵢ + xσ(i) +
β·yσ(i))`) over the columns `(node_idx, parent_idx, sibling_pos)`.

The *root commitment* exposed as a public input,
`subtree_root_of_root := W[0].subtree_root` (with `parent_idx = n`,
indicating the root), pins the entire tree shape to a single 255-bit
field element. Tampering with any single row (omitting a forbidden
node, swapping parent edges, fabricating a subtree) cascades into a
Poseidon hash mismatch and the verifier rejects.

**3. Range / domain gate — "no out-of-grammar nodes"** (the no-impossible-kind proof):

A simple range check `r.node_kind < MAX_KIND_FOR_LANGUAGE` enforced
via a 16-bit decomposition gate. In Halo2:

```
let bits = (0..16).map(|i| {
    meta.query_advice(node_kind_bit_cols[i], Rotation::cur())
}).collect::<Vec<_>>();
meta.create_gate("node_kind_range", |meta| {
    let kind = meta.query_advice(node_kind_col, Rotation::cur());
    let reconstructed: Expression<Fp> = bits.iter().enumerate()
        .fold(Expression::Constant(Fp::zero()), |acc, (i, b)| {
            acc + b.clone() * Expression::Constant(Fp::from(1u64 << i))
        });
    let bool_constraints: Vec<Expression<Fp>> = bits.iter()
        .map(|b| b.clone() * (Expression::Constant(Fp::one()) - b.clone()))
        .collect();
    let mut gates = vec![kind - reconstructed];
    gates.extend(bool_constraints);
    gates
});
```

This prevents the prover from injecting node-kind values outside the
language's tree-sitter grammar enum (which would otherwise let them
dodge the lookup gate by using a sentinel kind not in `F` ∪ `T_allowed`).

**Public inputs (the verifier sees only these)**:

```rust
pub struct ZkAttestationPublic {
    /// BLAKE3(file_path) ⊕ Poseidon(subtree_root_of_root)
    pub codebase_root: [u8; 32],
    /// Poseidon hash of (rule_id, forbidden_node_kinds, language_id)
    pub rule_set_hash: [u8; 32],
    /// BLAKE3 of the engine binary
    pub engine_hash: [u8; 32],
    /// Verdict: 1 = no forbidden nodes present; 0 = present (proof fails)
    pub verdict: Fp,
}
```

The auditor verifies in `O(log n)` (Halo2 recursion) without seeing
`W`, the source bytes, or any AST structural detail beyond the root
commitment. Two codebases producing identical root commitments are
provably structurally identical (modulo Poseidon collision resistance,
which reduces to the discrete-log assumption on Pasta).

**Recursion / aggregation**: large codebases (n > 2^17 nodes) are
sharded; each shard produces a Halo2 proof; shards are recursively
folded via the Halo2 accumulator-IPA scheme (`halo2_proofs::recursion`).
A repository-level proof is a single 4 KiB blob attesting absence of
forbidden nodes across every file. Verification cost is constant in
shard count.

**Extensions for non-trivial rules**:

* **Counted-bound rule** (e.g. "no more than 3 unsafe blocks"): replace
  the lookup gate with a *grand-sum* gate computing
  `Σᵢ [r_i.node_kind == UNSAFE_BLOCK]` and assert the sum ≤ 3 via a
  range gate.
* **Pair predicate** (e.g. "no `eval` whose argument is a function
  parameter"): two coupled lookup gates over `(node_kind, parent_kind,
  arg_origin_label)` — requires lifting the IFDS taint label into the
  witness table (extends seamlessly: `r.taint_label : Fp = Poseidon(label_string)`).
* **Reachability** (e.g. "no source flows into sink"): a permutation
  argument over the IFDS `(seed_fact, summary_chain, reached_sink)`
  triples; reuses the same Poseidon Merkle infrastructure with a
  `IfdsTraceRow` extension.

**Plonky3 alternative**: for environments where halo2's recursion stack
is unavailable, Plonky3 with the Goldilocks field, the same constraint
system, and FRI-based commitments produces faster proofs (≈3× over
halo2 on commodity hardware) at the cost of larger proof size
(≈45 KiB vs 4 KiB). The choice is operator-driven via
`ForgeConfig::zk_backend = { Halo2 | Plonky3 }`.

**Crates** (specific): `halo2_proofs = "0.3"`, `halo2_gadgets = "0.3"`
(Poseidon chip), `pasta_curves = "0.5"` (Pallas/Vesta IPA);
alternatively `plonky3 = "0.1"` with `p3-goldilocks`, `p3-poseidon2`,
`p3-fri`. Crate layout:

```
crates/forge/src/zk_attestation/
├── mod.rs              # public API
├── ast_witness.rs      # AST → AstRow witness lift
├── circuit.rs          # Halo2 ConstraintSystem definition
├── poseidon_chip.rs    # in-circuit Poseidon (or use halo2_gadgets)
├── recursion.rs        # shard accumulator
└── plonky3_backend.rs  # alternative backend
```

**Performance budget** (engineering targets):

* Witness generation: ≤ 50 ms / 2^17-node shard on M2 Pro.
* Proving time: ≤ 8 s / shard (halo2 IPA) or ≤ 2 s / shard (Plonky3 FRI).
* Verification: ≤ 5 ms / proof, regardless of shard count after recursion.
* Proof size: ≤ 4 KiB (halo2 IPA) or ≤ 45 KiB (Plonky3).

**Determinism**: Poseidon round constants are baked into the crate
(constant-time table); IPA / FRI commitments use a transcript seeded by
`BLAKE3(rule_set_hash || engine_hash)` so the proof is reproducible
bit-for-bit across runs given identical inputs.

**Why this beats the competitive landscape**: no SAST competitor today
ships a circuit-level absence proof. CodeQL, Semgrep, Snyk, Checkmarx
all require source upload to verify their findings. The construction
above lets an auditor verify "this codebase has no `eval` calls" in
milliseconds without ever seeing the source — the canonical zero-upload
unlock for FedRAMP High, DoD IL6, and EU CRA Article 11 attestation.

### P5-6 — Zero-Knowledge Exploit Brokerage \& On-Chain Bounty Settlement [DEFERRED to Sprint 71]

**The gap**: The bug bounty market (HackerOne / Bugcrowd / YesWeHack / Immunefi) is a manual-trust-anchored marketplace — the hunter trusts the platform to triage fairly, the vendor trusts the platform to obfuscate the report from public view. Both sides eat 20-30% in platform fees and 4-12 weeks of latency. The cryptographic moat: zk-SNARK proof-of-exploit + on-chain escrow that pays out on auditor confirmation, with NO platform middleman.

**Build**:

1. **Proof-of-exploit circuit** (`crates/forge/src/zk\_exploit\_proof.rs` — new module): for each Critical-severity finding with an `ExploitWitness` containing a satisfiable Z3 model, compile the model + the source-to-sink IFDS path into an arithmetic circuit (Plonky3 or Halo2). The public inputs commit to: `(codebase\_root\_hash, rule\_id, severity)`. The proof attests "I ran the engine on a codebase whose merkle-root is X and produced a Critical finding under rule R" without revealing the source.
2. **On-chain escrow contract** (Solidity / Move / Cairo): the vendor deposits the bounty amount; the hunter submits the zk-proof; the contract verifies the proof on-chain (via a precompiled verifier — Plonky3 verification is \~50ms in EVM); on success, payout is automatic and atomic. Pairs with P5-1 (Zero-Knowledge Attack Surface Attestation) for the underlying circuit pipeline.
3. **Auditor handoff**: the contract supports an optional auditor-confirmation tier — for high-value bounties, an auditor signs a "confirmed exploitable in production" attestation; the contract releases the full bounty only after both the zk-proof AND the auditor signature are present.
4. **Sealed disclosure window**: the contract supports a 90-day private window — the proof is verifiable but the public inputs are encrypted to the vendor's pubkey. After 90 days, the vendor's decryption key is revealed and the public can audit. Mirrors responsible-disclosure norms while removing the platform middleman.
5. **Reputation-bonded zk-claims** (anti-spam): hunters stake an Ed25519 / ML-DSA-65 keypair-bound reputation token. Invalid claims slash the stake; valid claims compound it. Bonded staking eliminates the proof-spam griefing vector that any open marketplace must defend against.

* `BountyEscrow { vendor\_address, bounty\_amount, deadline, auditor\_pubkey: Option<\_>, claim: Option<ZkExploitClaim> }`

### P6-6 — LLM-Agent Decompilation: Adversary Memory \& Tool-Catalog Reverse Engineering

**The gap**: A malicious or compromised LLM agent (Mythos / Kimi / autonomous red-team frameworks) is a black box at runtime. The agent's "memory" — its system prompt, tool catalog, fine-tune deltas, retrieval index — encodes its capabilities and intent. Today, defenders cannot extract these without source. Decompilation lets the defender reconstruct the agent's actual behavior surface from observable artifacts (binary weights, API traces, tool definitions).

**Build**:

1. **Tool-catalog static decompiler** (`crates/forge/src/llm\_decompile.rs` — new module): parse OpenAI function-calling schemas, MCP server tool definitions, LangChain `Tool` registrations, AutoGen `register\_for\_llm` annotations, Anthropic tool-use schemas. Emit a unified `AgentToolCatalog { agent\_id, tools: Vec<DecompiledTool> }`.
2. **System-prompt extractor**: many agentic frameworks ship the system prompt as a hardcoded string literal or a templated JSON file. Extract via the existing tree-sitter + AhoCorasick stack; emit `SystemPromptCapsule { agent\_id, prompt\_text, sha384 }`. Cross-reference against a corpus of known-malicious prompt patterns.
3. **Fine-tune delta inference**: when a model ships as a base + LoRA adapter, parse the LoRA weight deltas (`.safetensors` with `lora\_A`, `lora\_B` tensors). Compute statistical signatures (rank, sparsity, neuron-class deviation from base) — emit `FineTuneFingerprint { base\_model\_hash, delta\_signature, anomaly\_score }`.
4. **Behavior surface synthesis**: combine `(tool\_catalog, system\_prompt, fine\_tune\_fingerprint)` into a `BehaviorSurface` capsule. Static analysis on the capsule emits findings of class:

   * `security:agent\_lateral\_capability` — tools that compose into LPE / data-exfiltration chains.
   * `security:agent\_intent\_misalignment` — system prompt claims X, tools enable Y and reuses the shipped intent-divergence guard.
   * `security:agent\_provenance\_break` — LoRA delta cannot be matched to a public, signed adapter.
5. **Determinism**: all decompilation is reproducible — fixed-seed tokenizer, bounded statistical analysis, no online model invocation.

**Crates**: existing tree-sitter (JS/TS/Python/Go/Rust); `safetensors` (existing); `candle-core` (bounded, offline); existing IFDS engine; existing rkyv catalog.

**Deliverables**:

* `crates/forge/src/llm\_decompile.rs` — tool-catalog + system-prompt + LoRA delta extractor

### P6-8 — Steganographic Binary Carrier Detection (PureRAT Class)

**The gap**: PureRAT-class loaders embed multi-megabyte base64 / hex string literals inside Python / JS / Ruby / PHP modules. The literals decode to Windows PE or Linux ELF binaries; at runtime the module decodes and executes via `ctypes`, `os.exec`, in-memory loaders, or hollowed processes. No SAST vendor today decodes string literals.

**Build**:

1. **Long-literal scanner** (`crates/forge/src/stego\_binary.rs` — new module): for any string literal ≥ 4 KiB, attempt:

   * Base64 decode (RFC 4648 standard + URL-safe alphabet).
   * Hex decode.
   * Decompress (deflate / gzip / lzma) — bounded 32 MiB.
2. **Binary header recognition**: if decoded bytes start with `MZ` (PE), `\\x7fELF` (ELF), `\\xca\\xfe\\xba\\xbe` (Mach-O fat), `\\xfe\\xed\\xfa\\xce` / `\\xfe\\xed\\xfa\\xcf` (Mach-O 32/64), `MSCF` (CAB), or contain a valid PE / ELF / Mach-O section table (validated via existing `goblin`), emit `security:embedded\_executable\_blob` at `KevCritical`.
3. **AhoCorasick fast-path**: pre-filter via byte-pattern check for base64/hex alphabets in long contiguous runs; only invoke decode on candidate literals to keep cost bounded. Budget caps: decode at most 64 MiB per file, scan at most 50 long literals per file.
4. **Sink correlation** (IFDS): the long literal's variable identifier flows into `ctypes.CDLL`, `Function`, `eval`, `vm.Script`, `kernel32!CreateFileMapping`, `memfd\_create`, `dlopen("/proc/self/fd/...")`, or process-hollowing primitives — upgrade to `security:in\_memory\_loader`.
5. **Determinism**: all stages are deterministic; budget caps prevent DoS; no external network calls.

**Crates**: `base64` (workspace); `hex` (workspace); `flate2` for compression unwrap; `goblin` (existing) for PE/ELF/Mach-O header validation.

**Deliverables**:

* `crates/forge/src/stego\_binary.rs` — long-literal scanner + binary header recognition
* Crucible fixture: a Python file with a 1 MiB base64 literal that decodes to a stub PE — detector emits `embedded\_executable\_blob`. Negative fixture: a 100 KiB legitimate base64-encoded SVG asset — detector MUST NOT fire.

**Bounty TAM**: $50k–$200k per advisory; high-value class because no SAST vendor today decodes string literals. References PureRAT threat profile in `tools/campaign/ATTACK\_LEDGER.md`.

### P6-12 — Training-Data Trojan PR Detector

**The gap**: Open-source training datasets accept community PRs adding
new samples or correcting labels. An attacker submits a benign-looking
PR adding 200 trojan samples to a 2M-sample corpus. After the next
training cycle, the resulting model emits attacker-controlled output
when a trigger appears at inference. The poisoned model ships into
every downstream consumer of the dataset (TrojaNet / BadNets class).

**Build**:

1. **Dataset PR scanner** (`crates/forge/src/dataset_poisoning.rs` —
   new module): intercept PR diffs against directories matching
   `**/data/**`, `**/datasets/**`, `**/train/**`, `**/corpus/**`,
   `**/instruct/**`, `**/sft/**`; against extensions `.parquet`,
   `.arrow`, `.tfrecord`, `.jsonl`, `.csv`, `.feather`, `.npz`, `.h5`,
   `.safetensors`, `.zarr`.
2. **Format-aware sample extraction**: parse Parquet (`parquet 53`),
   Arrow (`arrow 53`), JSONL (`serde_json`), TFRecord (hand-rolled),
   NumPy `.npz` (`npyz 0.8`). Emit per-sample
   `(sample_idx, content_bytes, added_in_pr: bool)`.
3. **Text-trigger detection**:
   * Rare-Unicode block runs (`U+E0000`–`U+E007F`,
     `U+200B`–`U+200D`, `U+FEFF`) length ≥ 4 in any sample.
   * Repeated identical token sequences (length ≥ 8) appearing in ≥
     0.1% of newly added samples — statistically anomalous against
     base-rate distribution.
   * High-entropy base64/hex strings of length ≥ 256 bytes.
4. **Image-trigger detection**:
   * Deterministic perceptual hash (`pHash`, `dHash`) on each new
     sample; cluster by hash distance. Clusters of size ≥ 8 within
     added samples (but not pre-existing) are anomalous.
   * 4×4 corner-watermark detection: `for x in 0..4 for y in 0..4:
     pixel(x, y) ∈ trigger_palette` test.
5. **Distribution divergence**: KL divergence between added-samples
   class distribution and pre-existing class distribution. Divergence
   ≥ 0.5 nats on a labeled corpus emits
   `security:dataset_class_shift`.
6. **Sink emit**: `security:training_data_trojan` at KevCritical with
   affected sample indices, trigger-class, and a Crucible-friendly
   reproduction recipe.

**Determinism**: pHash/dHash use fixed-seed DCT; entropy windows are
fixed-width; AhoCorasick patterns rkyv-baked.

**Crates**: `parquet 53`, `arrow 53`, `npyz 0.8`, existing
`aho-corasick`, `image 0.25`, `image_hasher 1.2`.

**Deliverables**:

* `crates/forge/src/dataset_poisoning.rs` — multi-format scanner +
  trigger detector + distribution divergence
* `crates/anatomist/src/dataset_ingest.rs` — Parquet/Arrow/TFRecord
  parser bridge
* Crucible fixture: PR adding 200 JSONL samples each containing
  `\u{E0073}\u{E0079}\u{E0073}\u{E0074}` — fires. PR with 200 normal
  samples within 0.05 nats KL divergence — no fire.

**Bounty TAM**: $50k–$500k per advisory; the canonical 2026
ML-supply-chain class. Captures the AI/ML procurement budget that no

## Architectural Invariants (Non-Negotiable Across All Phases)

1. **Determinism is sacred.** Every analysis path must produce identical
results given identical inputs. No floating-point non-determinism, no
thread-ordering sensitivity, no wall-clock-dependent decisions. ML
ranks; rules decide.
2. **Zero-upload guarantee.** Source code never leaves the device. No
cloud API calls for analysis. The Janitor is air-gap-capable today and
must remain so through every phase, including P5 (ZK / FHE / iO / TEE
channels emit only attestations, never source).
3. **Bounded resource consumption.** Every new analysis phase must
respect the Physarum backpressure system and the 8 GB Law. Symbolic
execution gets a 10,000-state budget. IFDS gets a per-function summary
cache cap. Genetic rule evolution gets a bounded generation count.
Unbounded analysis is denial-of-service against the operator.
4. **Provenance continuity.** Every finding, at every phase, is sealed
into a `DecisionCapsule` with analysis parameters, engine version,
taint catalog hash, rule set hash, and Wasm policy receipts. A finding
without provenance is not a finding.
5. **Fail-open on analysis, fail-closed on trust.** A parse failure,
solver timeout, or budget exhaustion produces a degraded result (fewer
findings), never a crash or a false "all clear." The base
pattern-matching tier is always available as a floor. Conversely, any
trust-anchor failure (PQC signature invalid, enclave quote rejected,
rule lineage broken) is a hard-stop — trust never degrades gracefully.
6. **Structural gate mandate.** Every new phase that detects a
vulnerability must also contribute a Crucible fixture (true-positive
* true-negative) so the detection is permanently regression-guarded.
No phase ships without its Crucible entry.

### P18-2 - Chronometric Signature Split-Brain

**The gap**: Distributed auth systems rely on local clocks for JWT, SAML,
webhook, signed URL, HMAC, and nonce expiry decisions. Clock skew can create
acceptance windows where one verifier rejects a message while another accepts
and forwards it into a privileged path.

**Build**: Add `crates/forge/src/chronometric_auth.rs`. Extract timestamp
validation predicates, allowed skew constants, replay-cache scopes, and
cross-service forwarding edges. Emit `security:clock_skew_auth_split_brain`
when the composed verifier graph admits inconsistent acceptance.

**Rust/Z3 mathematics**: encode service clocks as integer variables
`t_i = t_real + skew_i`, with bounds from config. For each verifier `v_i`,
assert `nbf <= t_i + leeway_i <= exp` plus nonce-domain constraints. Z3 solves
for `Accept(A) && Reject(B) && Forward(A, B)` to prove split-brain reachability.

**Commercial unlock**: Covers webhook security, financial APIs, SSO, and
multi-region SaaS control planes where ordinary taint analysis has no temporal
model.

### P18-4 - Embedding-Manifold Trust Transposition

**The gap**: RAG systems can move untrusted documents into trusted answer space
when embeddings place attacker-controlled chunks near internal policy,
support, or runbook content. The bug is not string injection; it is trust-label
movement through vector geometry.

**Build**: Add `crates/forge/src/embedding_trust.rs`. Track document provenance,
embedding model id, vector-store namespace, and retrieval thresholds. Emit
`security:embedding_trust_transposition` when an untrusted vector can dominate
trusted context under the configured top-k and similarity metric.

**Rust/Z3 mathematics**: store normalized vectors as bounded `f32` slices,
quantize to fixed-point `i32`, and prove cosine-margin violations with integer
arithmetic: `dot(u, q) - dot(t, q) >= margin`. Z3 produces the violating
ranking relation; Rust verifies the same relation with saturating fixed-point
math.

**Commercial unlock**: AI governance budgets are moving from prompt filters to
provenance guarantees. This creates a deterministic RAG assurance SKU without
local LLM inference.

### P18-5 - Revocation-After-DMA Shadow Access

**The gap**: Hardware and driver stacks can revoke logical access while DMA,
shared buffers, or accelerator queues still hold usable references. The policy
state says access ended; the device can still observe or mutate memory.

**Build**: Add `crates/forge/src/dma_revocation.rs`. Parse driver code, eBPF
maps, VFIO/IOMMU setup, GPU queue submission, and shared-memory lifetimes.
Emit `security:dma_revocation_shadow_access` when a revocation event does not
dominate every outstanding DMA descriptor or mapped buffer.

**Rust/Z3 mathematics**: model buffer lifetimes as intervals
`[map_start, unmap_end]`, descriptor execution as `[submit, complete]`, and
revocation as `revoke_t`. Z3 checks `submit < revoke_t < complete` while Rust
dominators prove no `dma_unmap` or queue fence lies on every path before
revocation.

**Commercial unlock**: Opens hardware-software boundary assurance for fintech
HSM integrations, GPU AI clusters, EDR agents, and regulated endpoint fleets.

### P0-REV-1 — Bugcrowd Submission Formatter (Immediate Revenue)

**Strategic objective**: The engine already finds real vulnerabilities in
Bugcrowd-enrolled programs. The bottleneck is the manual step between
`janitor hunt` output and a submission-ready Bugcrowd report. Every hour saved
on formatting = more programs hunted per day = more revenue per week.

**The gap**: `janitor hunt --format bugcrowd` produces a Markdown report but it
does not (a) verify the finding is in-scope per the program's `_targets.md`
rules file, (b) auto-populate Bugcrowd's required submission fields (Title,
CVSS, Reproduction Steps, Impact, Remediation), or (c) warn when a finding's
severity exceeds the program's payout ceiling (avoiding wasted effort on out-of-
scope submissions).

**Build**:

1. Add `crates/cli/src/submit_formatter.rs`. Read the program's
   `tools/campaign/targets/<program>_targets.md` and extract the scope rules,
   payout table, and out-of-scope exclusions via AhoCorasick.
2. For every finding in the hunt output, cross-reference its file path and
   finding class against the extracted scope rules. Emit a `[SCOPE: IN]` /
   `[SCOPE: OUT]` verdict inline.
3. Auto-generate a `SUBMISSION.md` alongside each hunt report with Bugcrowd's
   required fields pre-filled from the finding's `remediation`, `docs_url`,
   `severity`, and `repro_cmd` fields.
4. Add `--submit-check` flag to `janitor hunt` that runs this formatter
   automatically after the scan completes.

**Crates**: `aho-corasick` (existing), `serde_json` (existing). Zero new
dependencies. 8GB Law compliant.

**True-positive fixture**: A finding in `tools/campaign/targets/` scope with
`repro_cmd` set → `SUBMISSION.md` generated with all fields populated.
**True-negative fixture**: A finding against an out-of-scope URL → `[SCOPE: OUT]`
warning emitted, no `SUBMISSION.md` generated.

**Revenue estimate**: Bugcrowd programs in the current ledger pay $100–$3,000

the operator can process 9× more findings per week. At one valid finding per

volume — this formatter multiplies the throughput of every future sprint.

**Time to first dollar**: 1–2 weeks from build completion (Bugcrowd review SLA

### P0-REV-2 — Immunefi Smart Contract Audit Lane (Highest Single-Report Payout)

**Strategic objective**: Immunefi is the highest-paying public bug bounty

hundreds of other Web3 protocols offering $10,000–$1,000,000+ per critical
vulnerability. The engine already has `crates/forge/src/solidity_taint.rs`.
The gap is that the existing detector is not tuned to Immunefi's Vulnerability
Classification System (VCS), which defines the exact severity thresholds and
exclusion classes that determine payout eligibility.

**The legal MEV clarification**: The operator asked about MEV as a revenue
path. There are two categories:
- **Legitimate MEV** (legal, protocol-permitted): price arbitrage between DEXes,
  protocol-defined liquidations (Aave, Compound — liquidators earn a 5–15% bonus
  for clearing undercollateralized loans). These require their own Rust bots and
  are highly competitive; expected monthly income is unpredictable and often near
  zero for a new entrant without co-location or block-builder access.
- **Exploit-based extraction** (not a bug bounty path): finding a re-entrancy
  or integer overflow in a live contract and submitting a transaction to drain
  it. This is a crime (Computer Fraud and Abuse Act, Fraud Act 2006) even if the
  contract has no formal owner. Do NOT pursue this path.
- **The correct path using this engine**: find the same re-entrancy or overflow
  *before it is exploited*, submit it to the protocol's Immunefi program, and
  collect the bounty legitimately. Immunefi's Critical tier pays $50k–$1M.
  This is identical to what the engine already does for web vulnerabilities —
  just in Solidity.

**Build**:

1. Extend `crates/forge/src/solidity_taint.rs` with Immunefi VCS rule mapping:
   - `security:reentrancy` → Immunefi Critical ($50k–$1M)
   - `security:integer_overflow` → Immunefi High ($10k–$50k)
   - `security:unsafe_delegatecall` → Immunefi Critical (code execution)
   - `security:unchecked_external_call` → Immunefi Medium ($1k–$10k)
2. Add `--format immunefi` output mode to `janitor hunt` that emits the
   Immunefi submission template (Title, Severity, Description, Proof of Concept,
   Recommended Fix, Impact).
3. Add a `tools/campaign/targets/immunefi_targets.md` program file with the 10
   highest-payout programs currently open on Immunefi, their scope URLs
   (GitHub repos), and their payout tables. These are publicly listed.
4. Add the Immunefi targets to `target_ledger.json`.

**Crates**: `solidity_taint.rs` (existing), `serde_json` (existing). No new
dependencies.

**True-positive fixture**: Solidity contract with `call.value()` before
`balances[msg.sender] = 0` (classic re-entrancy) → `security:reentrancy`
Critical emitted with Immunefi submission template populated.
**True-negative fixture**: Contract with checks-effects-interactions pattern
applied → no finding.

**Revenue estimate**: A single Immunefi Critical finding in a mid-tier program
