    Innovation Log — v10.1 Strategic Blueprint

    Generated 2026-04-10. Apex Market Audit. Authoritative roadmap for post-GA expansion. All prior entries purged.

    P0 — Enterprise Table Stakes

    P0-1: Signed Policy Lifecycle (janitor.toml Tamper Resistance) — [RESOLVED v10.1.0-alpha.1]

    • Class: Governance / Trust
    • Resolved: JanitorPolicy::content_hash() implemented in crates/common/src/policy.rs. Governor /v1/analysis-token enforces JANITOR_GOV_EXPECTED_POLICY env var; returns HTTP 403 policy_drift_detected on hash mismatch. policy_hash now computed from canonical struct fields (not raw TOML bytes) — resilient to whitespace/comment edits.

    P0-4: Role-Based Access Control (RBAC) for Suppressions

    • Class: Governance / Compliance
    • Observation: [[suppressions]] in janitor.toml can be authored by any committer. Enterprise buyers (SOC2 Type II, ISO 27001) require an audit trail showing that only authorised security leads can suppress findings.
    • Proposal: Suppressions require a Governor-side approval. The owner field in Suppression is verified against a Governor-managed ACL. Unapproved suppressions are ignored and flagged as unauthorised_suppression findings.
    • Implementation Path:
    • Governor API: POST /v1/approve-suppression (admin-only); stores (suppression_id, approver, approved_at).
    • crates/cli/src/main.rs: cmd_bounce sends suppression IDs to Governor; Governor responds with approval status per ID.
    • crates/common/src/policy.rs: Suppression gains approved: bool (runtime, not serialised in TOML).

    P0-5: Incremental / Resumable Scan — [RESOLVED v10.1.0-alpha.1]

    • Class: Scale / UX
    • Resolved: crates/common/src/scan_state.rs — ScanState { cache: HashMap<String, [u8; 32]> } with rkyv derives and symlink-safe atomic persistence. bounce_git in crates/forge/src/slop_filter.rs skips files whose BLAKE3 blob digest matches the cache (O(1) incremental bypass). State loaded before bounce and persisted after in cmd_bounce.

    P1 — Taint Spine Expansion

    P1-1: Deep Intra-File Taint for C#, Ruby, PHP, Swift, and the Remaining KEV Languages

    • Class: Detection Depth
    • Observation: Full intra-file SQLi/SSRF/path-traversal taint analysis exists for Python, JS/TS, Java, Go, and baseline C#. Interprocedural taint spine expansion is still required for C#, Ruby, PHP, Swift, Kotlin, Scala, Lua, Bash, Nix, GDScript, Objective-C, C/C++, GLSL, and Rust have AST antipattern walks but lack taint source→sink propagation. Semgrep's Feb 2026 release specifically improved taint tracking — this is the competitive battleground.
    • Proposal: Implement find_<lang>_sqli_slop + find_<lang>_ssrf_slop for the remaining grammars in priority order:
    • Ruby (Rails — massive install base, SQLi via where("#{}"))
    • PHP (WordPress/Laravel — largest CVE surface)
    • Kotlin (Android — JVM taint, Runtime.exec)
    • C/C++ (buffer overflow source→sink: gets/strcpy/sprintf → stack)
    • Swift (iOS — URLSession SSRF, FileManager traversal)
    • Implementation Path:
    • crates/forge/src/slop_hunter.rs: new find_ruby_sqli_slop(), find_php_sqli_slop(), etc. using tree-sitter queries.
    • crates/forge/src/taint_propagate.rs: new collect_ruby_params(), find_tainted_ruby_sql_sinks(), etc.
    • crates/crucible/src/main.rs: 2 entries per language (true-positive + true-negative).

    P1-2: Cross-File Taint — Remaining 18 Grammars

    • Class: Detection Depth
    • Observation: Cross-file taint (taint_catalog.rs) covers Python, JS, Java, TS, and Go. The remaining 18 grammars have no cross-file taint at all. Checkmarx's primary differentiator is cross-file/cross-method taint.
    • Proposal: Extend scan_cross_file_sinks dispatch to Ruby, PHP, C#, Kotlin, C/C++, Rust, Swift, Scala (8 highest-CVE-density languages).
    • Implementation Path:
    • crates/forge/src/taint_catalog.rs: new scan_ruby(), scan_php(), scan_csharp(), scan_kotlin(), scan_cpp(), scan_rust(), scan_swift(), scan_scala() functions.
    • Each follows the established pattern: walk call nodes, match against CatalogView exported taint records, emit CrossFileSinkFinding.

    P1-3: Bitbucket Build Status API Publisher

    • Class: SCM Integration
    • Observation: GitHub (annotations), GitLab (Commit Status API), and Azure DevOps (Git Status API) have native verdict publishing. Bitbucket has env detection and stderr annotation only — no native API call. Bitbucket Cloud and Bitbucket Server are significant in enterprise (Atlassian stack).
    • Proposal: Implement BitbucketStatusPublisher::publish_verdict using the Bitbucket Build Status REST API (POST /2.0/repositories/{workspace}/{slug}/ commit/{sha}/statuses/build).
    • Implementation Path:
    • crates/common/src/scm.rs: ScmContext::from_pairs captures BITBUCKET_ACCESS_TOKEN; BitbucketStatusPublisher::publish_verdict POSTs to the Build Status API.
    • 1 new unit test: bitbucket_context_captures_api_credentials.

    P1-4: Marketplace Extensions (GitHub App / GitLab Integration / AzDO Extension)

    • Class: Distribution / GTM
    • Observation: Our GitHub integration is a composite Action. Competitors (Snyk, Semgrep, Checkmarx) distribute as GitHub Apps with one-click install, marketplace listings, and RBAC scoping. GitLab has a native integration directory. Azure DevOps has the VS Marketplace.
    • Proposal: Phase 1 — GitHub App marketplace listing (we already have the App; it needs a marketplace page with usage limits / billing). Phase 2 — GitLab CI template published to GitLab's template gallery. Phase 3 — Azure DevOps extension wrapping our action.yml logic.
    • Implementation Path:
    • Governor: implement OAuth callback for GitHub App installation events.
    • tools/gitlab-ci-template.yml: reusable .gitlab-ci.yml include.
    • tools/azdo-extension/: Azure DevOps extension manifest + task wrapper.

    P2 — The Unfair Advantage

    P2-1: Agentic Supply Chain Worm Detector (Sha1-Hulud Defense)

    • Class: AI Supply Chain / Zero-Day
    • Observation: The Sha1-Hulud npm worm (2025–2026) self-propagates by stealing npm tokens, enumerating publishable packages, and republishing infected versions. Semgrep responded with MCP server integration and agent swarms for TTP hunting. Current Janitor detection covers postinstall script hooks and credential entropy — but not the self-propagation pattern (version bump + npm publish in lifecycle scripts).
    • Proposal: Implement a detect_supply_chain_worm gate in slop_hunter.rs that fires at KevCritical severity when a package.json diff introduces: (a) npm publish or npm token in lifecycle scripts, (b) version bump + new postinstall/preinstall in the same diff, (c) obfuscated require chains resolving to child_process.exec.
    • Implementation Path:
    • crates/forge/src/slop_hunter.rs: new find_npm_worm_slop(source) AhoCorasick + AST walk.
    • crates/forge/src/metadata.rs: package_json_lifecycle_audit() extracts and classifies lifecycle script payloads.
    • crates/crucible/src/main.rs: Sha1-Hulud true-positive fixture (version bump + npm publish in postinstall) + true-negative (normal postinstall).

    P2-2: LLM Hallucination Firewall (Phantom Dependency Detector)

    • Class: AI Supply Chain / Slopsquat
    • Observation: LLMs hallucinate package names that don't exist on registries. Attackers register these hallucinated names and publish malware (slopsquatting). We already detect known slopsquat imports in Python/JS/Rust via corpus match. The corpus (43 entries) is tiny. Semgrep's supply chain tool uses the OSV.dev malicious package database.
    • Proposal: Expand the slopsquat corpus to 500+ entries by ingesting the OSV.dev malicious-packages dataset. Add a janitor update-slopsquat command that fetches the latest OSV malicious-package advisories and rebuilds the AhoCorasick automaton.
    • Implementation Path:
    • crates/cli/src/main.rs: cmd_update_slopsquat fetches OSV API (/v1/query with ecosystem filter for npm, PyPI, crates.io).
    • crates/forge/src/slop_hunter.rs: slopsquat corpus loaded from .janitor/slopsquat_corpus.rkyv (rkyv zero-copy) instead of hardcoded array.
    • crates/common/src/wisdom.rs: SlopsquatCorpus struct with rkyv derives.

    P2-3: Wasm Rule Marketplace

    • Class: Platform / Ecosystem
    • Observation: We have BYOP Wasm rule execution with fuel/memory bounds, BLAKE3 integrity pinning, and epoch timeout. No competitor offers this. But the rules must be authored from scratch — no sharing ecosystem exists.
    • Proposal: Create a curated rule registry (rules.thejanitor.app) where verified publishers can upload signed .wasm rules with metadata (language, CVE class, severity). CLI gains janitor rule install <rule-id> that fetches, verifies BLAKE3 + Ed25519, and writes to .janitor/rules/.
    • Implementation Path:
    • Governor API: GET /v1/rules (catalog), GET /v1/rules/{id}/download.
    • crates/cli/src/main.rs: cmd_rule_install, cmd_rule_list.
    • crates/vault/src/lib.rs: Ed25519 signature verification for rule packages.
    • Web: static site at rules.thejanitor.app with rule metadata + download links.

    P2-4: Real-Time Taint Visualization (IDE Extension)

    • Class: Developer Experience
    • Observation: Checkmarx and Snyk have VS Code extensions showing taint paths inline. Our TUI dashboard exists but is post-hoc. Real-time feedback during development catches issues before commit, not after push.
    • Proposal: Build a VS Code extension using LSP that runs janitor bounce on file save and renders inline taint annotations. Use the existing MCP server as the communication layer.
    • Implementation Path:
    • tools/vscode-extension/: TypeScript extension, LSP client.
    • crates/mcp/src/lib.rs: new janitor_lint_file tool (single-file bounce returning SARIF-like finding array).
    • Extension renders diagnostics from MCP response.

    P2-5: SBOM Drift Monitor

    • Class: Compliance / Continuous
    • Observation: We generate CycloneDX v1.6 CBOMs at bounce time. But SBOMs are point-in-time snapshots. Between bounces, dependencies can change (lockfile updates, transitive upgrades). No competitor offers continuous SBOM drift detection.
    • Proposal: janitor watch-sbom daemon mode that monitors Cargo.lock / package-lock.json / poetry.lock for changes via inotify/kqueue, re-generates the CBOM diff, and emits a webhook if new components appear that weren't in the last attested CBOM.
    • Implementation Path:
    • crates/cli/src/daemon.rs: SbomWatcher using notify crate (already a transitive dep via watchexec patterns).
    • crates/anatomist/src/manifest.rs: diff_sbom() comparing two DependencyRegistry snapshots.
    • Webhook: uses existing WebhookConfig with event type sbom_drift.

    P2-6: Post-Quantum Certificate Transparency for Wasm Rules

    • Class: Cryptographic Assurance
    • Observation: We have ML-DSA-65 + SLH-DSA-128s dual-PQC for release binaries and CBOMs. Wasm rules use BLAKE3 pinning only. A compromised rule registry could serve a valid-hash rule that was replaced between pin creation and execution.
    • Proposal: Extend PQC signing to Wasm rule packages. Each rule .wasm is accompanied by a .wasm.sig (ML-DSA-65 detached signature). The CLI verifies the signature using the rule publisher's public key before loading.
    • Implementation Path:
    • crates/forge/src/wasm_host.rs: WasmHost::load_module gains optional sig_path + publisher_key verification before Module::new.
    • crates/common/src/pqc.rs: verify_wasm_rule_signature().
    • janitor.toml: [[wasm_publishers]] table with publisher name + ML-DSA verifying key.
