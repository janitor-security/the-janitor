    P0 — Core Security

    2026-04-09 — GA Teardown Audit (Hostile Product Review)

    Verdict: not a toy engine; still not frictionless GA product for a 10,000-developer org.

    Already present — do not waste cycles re-building these: - SARIF output exists (janitor report --format sarif) and GitHub SARIF upload wiring already exists. - CLI progress spinners already exist for scan / clean flows; the gap is long-run orchestration and resumability, not absence of all progress UI. - Suppression primitives exist (# noqa-style pragmas, test-scope routing, local-brain janitor pardon); the gap is governable, reviewable suppression lifecycle.

    P0 — Enterprise Deployment Blockers

    P0-1 — No repository-governed suppression / waiver system

    Gap: Current suppression is split across inline pragmas and the local .janitor/local_brain.rkyv pardon model. That is not auditable policy. A 10,000-developer org needs reviewable, committed waivers with rule id, path glob, owner, justification, expiry, and severity ceiling. Silent local pardons are unacceptable in regulated environments because they create analyst-specific blind spots with no chain of custody.

    Required change: 1. Add [[suppressions]] to janitor.toml with fields: id, rules, paths, expires, owner, reason, max_severity. 2. Emit suppression hits into the bounce log, SARIF suppressions block, and rendered reports. 3. Hard-fail expired suppressions in CI.

    Why this blocks GA: Without governed waivers, the product cannot survive a real audit committee or central AppSec rollout.

    P0-2 — No enterprise case-management surface

    Gap: The engine emits scores and receipts; it does not emit assignable remediation cases. There is no first-class owner routing, service/team metadata, SLA clock, risk acceptance state, or ticket sync. Large organizations do not buy another detector that dumps JSON and says "good luck."

    Required change: 1. Add optional owner mapping (CODEOWNERS, team map, or manifest routing) to every finding. 2. Add Jira / Azure DevOps / Linear export adapters with deterministic finding fingerprints. 3. Add "new vs existing vs waived" state so reruns do not reopen noise forever.

    Why this blocks GA: No workflow integration means security ops becomes a human copy-paste farm.

    P0-3 — No resumable hyper-audit / strike execution ledger

    Gap: The product can batch-audit and render ledgers, but there is no explicit checkpoint/resume protocol for multi-hour adversarial strikes. If a 3-hour hyper-audit dies at 92%, the operator should not restart from zero. Long enterprise scans need durable progress state, resumable shards, and deterministic partial outputs.

    Required change: 1. Persist shard checkpoints under .janitor/strikes/<run-id>/. 2. Resume by PR cursor / git OID boundary. 3. Emit machine-readable heartbeat and ETA snapshots.

    Why this blocks GA: Marathon audits without resumability are demo-grade, not fleet-grade.

    P0-4 — GitHub-heavy operational surface; non-GitHub enforcement is incomplete

    Gap: SCM normalization exists, but the commercial operating model still assumes GitHub App, GitHub PR checks, GitHub Actions, and GitHub release paths. Enterprises running GitLab, Azure DevOps, Bitbucket Server, or mixed estate need first-class check publishers, not just environment parsing.

    Required change: 1. Add provider-specific check/status publishers for GitLab MR, Azure DevOps PR, and Bitbucket. 2. Add SARIF upload helpers or native API adapters per provider. 3. Make the docs and SKU language stop implying GitHub-first operation if the product claims universal SCM support.

    Why this blocks GA: "Works outside GitHub" is not credible until enforcement surfaces exist outside GitHub.

    P1 — Product / UX / Commercial Defects

    P1-1 — Pricing pages contradict themselves

    Gap: README.md and docs/index.md sell Team as "up to 25 seats" and then immediately say "No per-seat limits." That is revenue-ops sloppiness visible to customers. Procurement will interpret this as either deceptive pricing or internal confusion.

    Required change: 1. Pick one truth: seat-capped Team or unlimited Team. 2. Normalize README, docs site, Lemon Squeezy copy, and Terms. 3. Add a pricing FAQ for OSS maintainer, startup, and enterprise procurement paths.

    P1-2 — OSS maintainer onboarding is still too doctrinal

    Gap: The engine is strong; the product voice still assumes the user already buys into the manifesto. A broke maintainer needs a 5-minute path to "what do I run, what does it output, how do I suppress one false positive, how do I wire CI without reading a constitution."

    Required change: 1. Add an "OSS fast path" doc with one local command, one CI snippet, one suppression example, and one SARIF upload example. 2. Ship starter janitor.toml profiles for library, app, monorepo, and security-sensitive repo. 3. Add a janitor init command that generates sane defaults.

    P1-3 — Wasm BYOR lacks operator ergonomics

    Gap: Integrity pinning now exists, but the operator experience is still primitive. There is no dedicated pin-generation CLI, no manifest helper, no pin rotation workflow, and no policy-level explanation when mixed CLI + manifest rules collide.

    Required change: 1. Add janitor wasm-pin <PATH>. 2. Add janitor wasm-verify <PATH> --expected <BLAKE3>. 3. Document precedence rules between janitor.toml and repeated --wasm-rules flags.

    P1-4 — Finding explainability is too terse for broad rollout

    Gap: The engine produces strong machine labels, but many findings still read like detector IDs plus short prose. Enterprise adoption improves when each finding carries "why flagged", "why dangerous", "how to remediate", and "how to suppress correctly" in a predictable schema.

    Required change: 1. Expand structured findings with remediation and suppression guidance fields. 2. Surface those fields in SARIF help URIs / markdown. 3. Add stable finding documentation URLs.

    P2 — Scale / Platform Expansion

    P2-1 — No first-class drift dashboard for policy health

    Gap: The ledger is strong after the fact. There is still no compact view showing suppression growth, noisy rules, top owners, stale waivers, or score drift across services.

    Required change: 1. Add a policy-health dashboard and JSON endpoint. 2. Track waiver counts, expired waivers, and hottest findings per team.

    P2-2 — No lightweight maintainer mode for tiny repos

    Gap: Free capability is generous, but the product packaging still feels enterprise-first. Small OSS maintainers need a "minimal noise" preset with lower ceremony, tiny config, and human-readable output defaults.

    Required change: 1. Add janitor init --profile oss. 2. Add compact text summaries optimized for solo maintainers. 3. Publish migration guidance from zero config to governed config.

    P1 — BYOP Supply Chain & Runtime Safety

    CT-022 — Wasm Rule Integrity Pinning (Deferred to v10.1)

    File: crates/common/src/policy.rs → ForgeConfig; crates/forge/src/wasm_host.rs → WasmHost::new

    Gap: The --wasm-rules <PATH> CLI flag loads arbitrary .wasm modules from the filesystem with no policy-pinned hash check. WasmPolicyReceipt.module_digest records the BLAKE3 of the loaded module after the fact, but there is no pre-load assertion against a known-good value. An attacker who can write to the wasm rule path (e.g., via a compromised build artifact store or path injection) can inject arbitrary Wasm that executes inside the 10 MiB sandbox against every source file analyzed. No SOC2 Type II, FedRAMP Moderate, or CMMC Level 2 auditor will approve the BYOP tier without this gate.

    Required change: 1. Add wasm_rule_pins: Vec<WasmRulePin> to ForgeConfig in policy.rs: 

    [[forge.wasm_rules]]
path = "rules/audit_rule.wasm"
blake3 = "e4cda1bb..."  # required; absent = hard-fail
    2. WasmHost::new() computes BLAKE3 of each loaded module bytes and asserts against the corresponding pin entry. Missing pin entry or hash mismatch = anyhow::bail!. 3. Add janitor wasm-pin <PATH> subcommand that prints the [[forge.wasm_rules]] TOML stanza. 4. Add Crucible entry: wasm_rule_hash_mismatch_rejected (true-positive) + wasm_rule_hash_match_accepted (true-negative).

    Commercial justification: Wasm rule pinning is the gate between "BYOP available" and "BYOP enterprise-approved". Every Fortune 500 running the Sovereign tier requires a cryptographically-bound bill of rules — analogous to SLSA Level 4 for rule artifacts. Unlocks the "Verified Rule Marketplace" pricing tier and directly answers the FedRAMP AU-9 audit control requirement for integrity of security tools. TAM expansion: +$30K/yr per Sovereign seat that activates custom rules.

    CT-023 — Wasm Epoch Thread Pool Leak (Deferred to v10.1)

    File: crates/forge/src/wasm_host.rs → WasmHost::run_module

    Gap: run_module() spawns a detached thread (via std::thread::spawn) per analysis invocation to fire the 100 ms epoch deadline. If the Wasm module completes before the timer, the thread is not cancelled — it sleeps, wakes, and calls engine.increment_epoch() on the shared engine. Under the gauntlet-runner 1000-PR strike with Wasm rules active, this creates up to concurrency × files_per_pr × wasm_modules simultaneous orphaned threads. Each thread consumes an 8 MiB stack on Linux by default. At 8 concurrent workers × 50 files × 1 Wasm module = 400 orphaned threads = 3.2 GiB stack space reserved — an OOM vector that bypasses the Physarum gate (which guards heap, not stack).

    Required change: 1. Add Arc<AtomicBool> cancellation flag, set to true after analyze_fn.call() returns. 2. Timeout thread checks the flag before calling increment_epoch(); if cancelled, exits immediately without touching the engine. 3. Alternatively, migrate to wasmtime's Store::set_epoch_deadline + a shared EpochTicker background thread that advances at 100 ms intervals globally (one thread total, not one per invocation). This is the architecturally preferred solution and reduces thread overhead from O(analyses) to O(1). 4. Add a unit test that verifies concurrent WasmHost::run() calls with 8 workers do not accumulate more than workers + 1 live threads.

    Commercial justification: Directly unblocks the gauntlet-runner adversarial strike feature at scale. Without the fix, enabling --wasm-rules on large monorepo strikes (10k+ files) risks OOM on the customer runner. The fix also eliminates the only remaining thread-count unboundedness in the engine — a finding that surfaces in every penetration test of the CI agent.
