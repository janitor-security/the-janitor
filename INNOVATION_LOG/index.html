    Critical Definitions

    Term Definition
    FIPS 140-3 boundary Any cryptographic operation whose output is used to make a security decision (auth, audit integrity, attestation)
    NIST-approved SHA-2 (SHA-256/384/512), SHA-3, HMAC-SHA-2, AES-GCM, ECDSA P-384, ML-DSA-65 (FIPS 204), SLH-DSA (FIPS 205)
    BLAKE3 status NOT NIST-approved. Acceptable for performance-only paths (AST deduplication, corpus dedup). BANNED at FIPS boundaries.

    P3 — Supply Chain & SBOM Compliance (FedRAMP SA / SR Controls)

    P3-2: Reproducible Build Attestation (SLSA Level 3 → Level 4)

    Status: OPEN — Upgrade from current SLSA Level 4 (binary provenance)

    Finding: Current SLSA provenance covers binary hash + PQC signature. FedRAMP High requires full build reproducibility: two independent builds from identical source must produce bit-identical binaries. Rust's opt-level = "z" with lto = true is currently NOT reproducible across different linker versions.

    Implementation plan: 1. Pin lld linker version in rust-toolchain.toml. 2. Set RUSTFLAGS="-C link-arg=-fuse-ld=lld -C link-arg=--build-id=none" in justfile::fast-release to strip build IDs from the output. 3. Add a just verify-reproducible recipe that builds the binary twice in isolated Docker containers and compares BLAKE3 hashes. 4. Emit in-toto provenance attestation using slsa-verifier format.

    2026-04-12 — Live ASPM Dedup Preflight Contract

    Status: OPEN

    Finding: The Jira deduplication proof path is not self-validating. bounce reaches KevCritical detection, but the live ASPM workflow aborts before dedup search/create when JANITOR_JIRA_USER or JANITOR_JIRA_TOKEN are absent from the execution environment. This leaves the operator with detector proof but no production deduplication proof.

    Implementation plan: 1. Add an explicit preflight to crates/cli/src/jira.rs or crates/cli/src/main.rs that reports the exact missing Jira auth variables before scan execution when [jira] is configured. 2. Add an integration harness that injects a deterministic mock Jira sender and asserts first-run create / second-run dedup skip semantics for a stable fingerprint. 3. Emit a dedicated terminal line for dedup preflight state so operators can distinguish detector success from ASPM transport readiness without inspecting warning text. 4. Add a Crucible or CLI regression covering the missing-auth fail-open path plus the successful dedup path.
