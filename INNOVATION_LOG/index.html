    Innovation Log — v10.1 Strategic Blueprint

    Generated 2026-04-10. Apex Market Audit. Authoritative roadmap for post-GA expansion. All prior entries purged.

    P0 — Enterprise Table Stakes

    P0-1: Signed Policy Lifecycle (janitor.toml Tamper Resistance) — [RESOLVED v10.1.0-alpha.1]

    • Class: Governance / Trust
    • Resolved: JanitorPolicy::content_hash() implemented in crates/common/src/policy.rs. Governor /v1/analysis-token enforces JANITOR_GOV_EXPECTED_POLICY env var; returns HTTP 403 policy_drift_detected on hash mismatch. policy_hash now computed from canonical struct fields (not raw TOML bytes) — resilient to whitespace/comment edits.

    P0-5: Incremental / Resumable Scan — [RESOLVED v10.1.0-alpha.1]

    • Class: Scale / UX
    • Resolved: crates/common/src/scan_state.rs — ScanState { cache: HashMap<String, [u8; 32]> } with rkyv derives and symlink-safe atomic persistence. bounce_git in crates/forge/src/slop_filter.rs skips files whose BLAKE3 blob digest matches the cache (O(1) incremental bypass). State loaded before bounce and persisted after in cmd_bounce.

    P1 — Taint Spine Expansion

    P1-2: Cross-File Taint — Remaining 18 Grammars — [RESOLVED v10.1.0-alpha.4]

    • Class: Detection Depth
    • Resolved: scan_cross_file_sinks dispatch extended with 8 new arms: Ruby, PHP, C#, Kotlin, C/C++, Rust, Swift, Scala. Each scanner walks call nodes using grammar-confirmed node types, emits CrossFileSinkFinding on catalog matches. 16+ true-positive/true-negative tests pass. Depth guards (>100) on all walkers.

    P1-3: Bitbucket Build Status API Publisher — [RESOLVED v10.1.0-alpha.4]

    • Class: SCM Integration
    • Resolved: ScmContext::from_pairs captures BITBUCKET_ACCESS_TOKEN, BITBUCKET_WORKSPACE, BITBUCKET_REPO_SLUG. BitbucketStatusPublisher::publish_verdict POSTs to https://api.bitbucket.org/2.0/repositories/{workspace}/{slug}/commit/{sha}/statuses/build with Bearer auth. Unit test bitbucket_context_captures_api_credentials added.

    P1-5: Full Taint Spine for Emerging Systems Languages (Zig, Nim)

    • Class: Detection Depth
    • Observation: We already have baseline AST parsing and dropper-pattern detection for emerging systems languages, but not full interprocedural taint propagation through their call graphs. That leaves a depth gap once those languages move from niche tooling into enterprise backend surfaces.
    • Proposal: Defer full taint-spine expansion for Zig and Nim until their backend adoption crosses critical mass, then extend the existing cross-file taint engine with grammar-native call/source/sink walkers instead of heuristic byte scans.
    • Implementation Path:
    • crates/forge/src/taint_catalog.rs: add Zig and Nim dispatch arms.
    • crates/polyglot/src/lib.rs: ensure grammar registry parity for both.
    • crates/crucible/src/main.rs: add true-positive and true-negative taint fixtures.

    P2 — The Unfair Advantage

    P2-3: Wasm Rule Marketplace

    • Class: Platform / Ecosystem
    • Observation: We have BYOP Wasm rule execution with fuel/memory bounds, BLAKE3 integrity pinning, and epoch timeout. No competitor offers this. But the rules must be authored from scratch — no sharing ecosystem exists.
    • Proposal: Create a curated rule registry (rules.thejanitor.app) where verified publishers can upload signed .wasm rules with metadata (language, CVE class, severity). CLI gains janitor rule install <rule-id> that fetches, verifies BLAKE3 + Ed25519, and writes to .janitor/rules/.
    • Implementation Path:
    • Governor API: GET /v1/rules (catalog), GET /v1/rules/{id}/download.
    • crates/cli/src/main.rs: cmd_rule_install, cmd_rule_list.
    • crates/vault/src/lib.rs: Ed25519 signature verification for rule packages.
    • Web: static site at rules.thejanitor.app with rule metadata + download links.

    P2-4: Real-Time Taint Visualization (IDE Extension)

    • Class: Developer Experience
    • Observation: Checkmarx and Snyk have VS Code extensions showing taint paths inline. Our TUI dashboard exists but is post-hoc. Real-time feedback during development catches issues before commit, not after push.
    • Proposal: Build a VS Code extension using LSP that runs janitor bounce on file save and renders inline taint annotations. Use the existing MCP server as the communication layer.
    • Implementation Path:
    • tools/vscode-extension/: TypeScript extension, LSP client.
    • crates/mcp/src/lib.rs: new janitor_lint_file tool (single-file bounce returning SARIF-like finding array).
    • Extension renders diagnostics from MCP response.

    P2-5: SBOM Drift Monitor

    • Class: Compliance / Continuous
    • Observation: We generate CycloneDX v1.6 CBOMs at bounce time. But SBOMs are point-in-time snapshots. Between bounces, dependencies can change (lockfile updates, transitive upgrades). No competitor offers continuous SBOM drift detection.
    • Proposal: janitor watch-sbom daemon mode that monitors Cargo.lock / package-lock.json / poetry.lock for changes via inotify/kqueue, re-generates the CBOM diff, and emits a webhook if new components appear that weren't in the last attested CBOM.
    • Implementation Path:
    • crates/cli/src/daemon.rs: SbomWatcher using notify crate (already a transitive dep via watchexec patterns).
    • crates/anatomist/src/manifest.rs: diff_sbom() comparing two DependencyRegistry snapshots.
    • Webhook: uses existing WebhookConfig with event type sbom_drift.

    P2-6: Post-Quantum Certificate Transparency for Wasm Rules

    • Class: Cryptographic Assurance
    • Observation: We have ML-DSA-65 + SLH-DSA-128s dual-PQC for release binaries and CBOMs. Wasm rules use BLAKE3 pinning only. A compromised rule registry could serve a valid-hash rule that was replaced between pin creation and execution.
    • Proposal: Extend PQC signing to Wasm rule packages. Each rule .wasm is accompanied by a .wasm.sig (ML-DSA-65 detached signature). The CLI verifies the signature using the rule publisher's public key before loading.
    • Implementation Path:
    • crates/forge/src/wasm_host.rs: WasmHost::load_module gains optional sig_path + publisher_key verification before Module::new.
    • crates/common/src/pqc.rs: verify_wasm_rule_signature().
    • janitor.toml: [[wasm_publishers]] table with publisher name + ML-DSA verifying key.
