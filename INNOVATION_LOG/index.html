    P2 — Cryptographic Assurance

    P2-6: Post-Quantum Certificate Transparency for Wasm Rules

    • Class: Cryptographic Assurance / DoD Compliance
    • TAM: DoD CMMC Level 2+ and FedRAMP High mandate FIPS 140-3 module integrity. Our release binaries are ML-DSA-65 + SLH-DSA-128s signed. Wasm rules use BLAKE3 pinning only. A compromised rule registry could serve a valid-hash rule that was replaced between pin creation and execution. Closing this gap is a hard requirement for any DoD/IC procurement.
    • Observation: crates/forge/src/wasm_host.rs loads .wasm modules from disk with BLAKE3 hash verification but no asymmetric signature check. The publisher's identity is unverified — any party who can replace the file and update the hash can inject arbitrary Wasm execution.
    • Proposal: Extend PQC signing to Wasm rule packages. Each .wasm is accompanied by a .wasm.sig (ML-DSA-65 detached signature). The CLI verifies the signature using the rule publisher's public key before loading.
    • Implementation Path:
    • crates/forge/src/wasm_host.rs: WasmHost::load_module gains optional sig_path + publisher_key verification before Module::new.
    • crates/common/src/pqc.rs: verify_wasm_rule_signature() using verify_asset_ml_dsa_signature with a JANITOR_WASM_RULE_CONTEXT domain separator.
    • janitor.toml: [[wasm_publishers]] table with publisher name + ML-DSA verifying key.
    • Crucible: wasm_rule_rejected_without_valid_pqc_signature entry.

    P1-2: OSV Bulk Corpus Resilience (Network Fault Tolerance)

    • Class: Threat Intel Infrastructure / Reliability
    • TAM: Live-fire audit (2026-04-11) confirmed update-slopsquat fails silently in air-gapped or network-restricted environments (WSL, corporate proxies, GCS-blocked egress): "reading OSV bulk ZIP response failed". No disk artifacts left (GC clean), but the command exits with an unrecoverable error and leaves the slopsquat corpus stale. Enterprises running in air-gapped CI have zero slopsquat protection after first install if the initial corpus fetch fails.
    • Observation: fetch_osv_slopsquat_corpus in crates/cli/src/main.rs makes a single synchronous HTTP request to osv-vulnerabilities.storage.googleapis.com. No retry, no fallback mirror, no embedded seed corpus. Failure leaves .janitor/slopsquat_corpus.rkyv absent rather than stale. The update-wisdom path has the same single-point-of-failure pattern against the Ed25519-signed wisdom URL.
    • Proposal: Add three-layer resilience:
    • Embedded seed corpus: bundle a curated slopsquat_corpus.rkyv snapshot in the binary (behind include_bytes!) as a last-resort fallback when network is unavailable.
    • Exponential backoff: 3 retries with 1 s / 2 s / 4 s delay before declaring failure.
    • Stale-corpus warning: if the corpus file is older than 7 days and a refresh attempt fails, emit [JANITOR DEGRADED] slopsquat corpus is stale to stderr and exit 0 (soft-fail) rather than exit 1, matching the soft_fail policy semantics already in JanitorPolicy.
    • Implementation Path:
    • crates/cli/src/main.rs: fetch_osv_slopsquat_corpus gains retry loop + embedded fallback.
    • crates/common/src/policy.rs: ForgeConfig gains corpus_stale_days: u32 (default 7).
    • build.rs (cli): include_bytes! of a baked seed corpus generated at release time.
